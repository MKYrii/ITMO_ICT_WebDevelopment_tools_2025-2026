## Задача 1

Цель работы: Сравнение производительности трех подходов к организации параллельных вычислений в Python (Threading, Multiprocessing, Asyncio) на задаче суммирования чисел в диапазоне от 1 до 1 000 000 000.

##### Реализация
Для всех трех подходов использовалась единая логика вычислений, реализованная в функции calculate_sum. Для обеспечения корректности сравнения общий диапазон чисел был разделен на 10 равных подзадач.

* Threading: задачи выполнялись в 10 потоках.
* Multiprocessing: вычисления распределялись между 10 независимыми процессами.
* Asyncio: использовался событийный цикл с выносом вычислений в потоки через to_thread и выполнением посредством asyncio.gather

#### Результаты замеров
При расчете суммы до $10^9$ с использованием 10 рабочих юнитов были получены следующие показатели времени:

| Подход | Время выполнения (сек) |
|---|---|
| Asyncio | ~28 секунд |
| Threading | ~28 секунд |
| Multiprocessing | ~4 секунды |

------------------------------
#### Анализ результатов
Разница в производительности обусловлена архитектурными особенностями интерпретатора CPython:

   1. Threading и Asyncio (~28 сек): Несмотря на распределение задач, оба подхода ограничены механизмом GIL (Global Interpreter Lock). Глобальная блокировка интерпретатора не позволяет выполнять Python-байткод на нескольких ядрах процессора одновременно. В итоге задачи выполняются поочередно на одном ядре. Относительно высокая скорость (по сравнению с глобальным циклом) достигнута за счет использования локальных переменных внутри функции calculate_sum, доступ к которым оптимизирован на уровне виртуальной машины Python.
   2. Multiprocessing (~4 сек): Данный модуль обходит ограничение GIL, создавая отдельные экземпляры интерпретатора для каждого процесса. Это позволяет распределить нагрузку между несколькими физическими ядрами процессора. Реальная параллельность вычислений обеспечила кратное ускорение, прямо пропорциональное количеству задействованных ядер.

#### Вывод

Для задач с высокой нагрузкой на процессор (CPU-bound) эффективным является только использование multiprocessing. Подходы threading и asyncio подходят для задач ожидания ввода-вывода (I/O-bound), но не дают реального ускорения в математических расчетах

## Задача 2

Цель работы: Сравнение производительности Threading, Multiprocessing и Asyncio при выполнении I/O-bound задачи: параллельного парсинга 15 веб-страниц с последующим асинхронным сохранением данных в реляционную базу данных PostgreSQL.

#### Реализация

Для заполнения базы данных был предварительно создан пользователь Parser. Информация, полученная в ходе парсинга страниц GitHub Issues, сохранялась в таблицу Task со следующими атрибутами: заголовок тикета (title), описание (description), временная метка (deadline) и идентификатор владельца (owner_user_id).

Для всех подходов использовалась единая функция парсинга на базе BeautifulSoup и aiohttp. 

Для обеспечения совместимости в подходах Threading и Multiprocessing использовалась синхронная функция-воркер, инкапсулирующая в себе запуск локального цикла событий (event loop) для выполнения асинхронных операций с БД

* Asyncio: 15 корутин выполнялись конкурентно в рамках одного событийного цикла с использованием asyncio.gather.
* Threading: создано 15 потоков, в каждом из которых запускался локальный цикл событий для выполнения асинхронных операций с БД и сетью.
* Multiprocessing: создано 15 независимых процессов. В каждом процессе инициализировался собственный движок БД (engine) и цикл событий.

#### Результаты замеров
При обработке 15 страниц GitHub Issues были получены следующие показатели времени:

| Подход | Время выполнения (сек) |
|---|---|
| Asyncio | ~1.28 секунд |
| Threading | ~1.35 секунд |
| Multiprocessing | ~3.20 секунд |

------------------------------
#### Анализ результатов
Результаты наглядно демонстрируют специфику задач, ограниченных вводом-выводом (I/O-bound):

   1. Asyncio и Threading (~1.3 сек): Оба метода показали высокую эффективность. В данной задаче основное время тратится на ожидание ответа от сервера GitHub и подтверждения транзакции от СУБД.
   * В Asyncio переключение между задачами происходит мгновенно внутри одного потока, что минимизирует накладные расходы.
   * В Threading операционная система переключает контекст между потоками, пока они заблокированы операциями ввода-вывода. Незначительное отставание потоков обусловлено затратами памяти на создание системных объектов Thread.
   2. Multiprocessing (~3.2 сек): Данный подход оказался наименее эффективным. Для каждого URL-адреса создавался отдельный процесс, что в ОС Windows сопряжено с копированием адресного пространства и инициализацией нового интерпретатора Python. Время, затраченное на "поднятие" 15 процессов, значительно превысило время выполнения самой полезной работы (парсинга и записи в БД).

#### Вывод
Для задач сетевого парсинга и взаимодействия с базами данных наиболее целесообразно использование Asyncio. Он обеспечивает максимальную производительность при минимальном потреблении системных ресурсов. Multiprocessing для подобных задач является избыточным, так как накладные расходы на управление процессами нивелируют выгоду от параллелизма.
